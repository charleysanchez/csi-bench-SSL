  ---
  Generalization Analysis

  1. Domain awareness barely helps (in-distribution)

  no_domain_aware ties overnight_v1 exactly on FallDetection (0.907) and is only
   0.6% behind on BreathingDetection. If domain contrastive learning were the
  main driver of generalization, you'd expect a larger gap even on test_id. The
  curriculum isn't hurting, but it's also not delivering a clear win.

  Implication: The benefit of domain awareness may only appear on OOD splits
  (new users/devices/environments) — which we don't see in these summary
  metrics. The test_id split is likely same-distribution held-out users.

  2. k_steps matters — but task-dependently

  - k_steps=8 dominates on FallDetection (+3.4% acc vs k_steps=4, same other
  configs)
  - k_steps=4 wins on BreathingDetection (+1% acc vs k_steps=8)

  FallDetection is a coarse, long-duration event — longer future context (8
  steps) captures temporal coherence across the fall arc. BreathingDetection is
  a periodic signal — shorter context (4 steps) may avoid aliasing or focus on
  local regularity. This task-specific optimum is a potential generalization
  hazard: a single k_steps may not generalize across task classes.

  3. Pretrain loss inverts downstream performance

  Baseline has the lowest pretrain loss (0.020) but worst downstream accuracy.
  This is the classic SSL sign that lower loss ≠ better representations —
  smaller emb_dim=256 + fewer negatives likely causes the model to fit easier
  shortcuts.

  4. Domain neg curriculum hasn't converged

  Target domain_neg_ratio was 0.8, but actual achieved by end of training:
  - overnight_v1: 0.66 (epoch 124)
  - k_steps_4: 0.69 (epoch 129)

  Neither reached the target. The 3.5h wall clock limit is cutting training
  short. The domain-aware curriculum is still ramping — you're measuring it
  before it has fully deployed domain negatives.

  5. Capacity matters: emb_dim 256→512 is worth it

  baseline vs k_steps_4 isolates emb_dim (256 vs 512), neg count (256 vs 512),
  and neg_ratio (0.5 vs 0.8). The larger config gains +3% Fall, +2.9% Breathing.
   For a heterogeneous deployment (new devices/environments), larger embeddings
  should capture more invariant features.

  ---
  Recommendations for Generalization

  Immediate:
  1. Add OOD eval to the probe — without cross-user/cross-device test splits,
  you can't actually measure generalization. test_id only shows in-distribution
  accuracy.
  2. Increase max_train_hours to 5-6h or add a floor on domain_neg_ratio to
  ensure the curriculum completes. The domain-aware benefit may be hidden
  because training ends before domain negatives dominate.

  Architectural:
  3. Task-adaptive k_steps: k_steps=8 for motion/event tasks, k_steps=4 for
  periodic tasks — or tune k_steps as a hyperparameter per task family rather
  than globally.
  4. overnight_v1 is the best single model (Fall+Breath balance, domain-aware,
  large emb) — use it as the reference going forward.

  Hypothesis to test:
  5. Does domain_aware gap vs no_domain_aware widen on OOD splits? If domain
  contrastive learning is doing its job, the 0.6% test_id gap should become
  3-5%+ on cross-device or cross-user splits. If it doesn't, the domain
  negatives aren't capturing the right invariance.
